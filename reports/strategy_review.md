## 1. 数据质量

### 1.1 [Critical] 严重前视偏差 — 回测使用当前快照模拟历史

**问题描述**：
`backtest.py` 的 `BacktestEngine.run()` 方法直接获取**当前**的强赎数据快照（`get_cb_redeem_data()`），然后用这个快照去匹配历史日线价格。这意味着：
- 当前快照中包含的转债状态（如"已公告强赎"）在历史回测日期可能尚未发生
- 回测中"买入"的转债，在买入时可能根本不存在或状态完全不同
- 这导致回测结果严重高估策略表现

**代码位置**：`backtest.py:148-155`
```python
# 获取最近一次强赎快照
redeem_df = get_cb_redeem_data()  # ← 获取的是当前数据！
# ...
scored = signal_rank(redeem_df, self.weights)  # ← 用当前数据评分
```

**修复建议**：
1. 改为逐日遍历回测：对每个历史日期，只使用该日期之前的数据
2. 或至少使用历史快照数据（如果akshare支持历史快照）
3. 实现一个`HistoricalDataProvider`类，按日期回放数据

**预期效果**：回测胜率预计下降30-50%，更接近真实表现

**工作量**：16-24小时

### 1.2 [High] 幸存者偏差 — 只包含当前存续转债

**问题描述**：
`get_cb_redeem_data()` 只返回**当前**有强赎状态的可转债，已经退市或完成强赎的转债不在数据中。回测时：
- 无法模拟"买入后转债退市"的场景
- 无法评估"买入后转债从未触发强赎"的失败案例
- 胜率被系统性高估

**代码位置**：`data.py:67-80`

**修复建议**：
1. 使用历史转债列表（akshare的`bond_zh_cov()`可获取历史数据）
2. 在回测中引入"退市风险"模拟
3. 至少记录并报告幸存者偏差的影响范围

**预期效果**：更真实的失败案例纳入评估

**工作量**：8-12小时

### 1.3 [Medium] 数据源单一且无校验

**问题描述**：
所有数据依赖单一的akshare数据源，且：
- 无数据质量校验（如价格异常值检测、缺失值处理）
- 无数据源切换/降级机制
- 无数据新鲜度检查

**代码位置**：`data.py` 全局

**修复建议**：
1. 增加数据质量检查函数：价格范围校验、缺失率统计
2. 实现数据源降级（如akshare失败时尝试本地缓存）
3. 增加数据新鲜度监控

**预期效果**：提高系统稳定性，减少因数据问题导致的误判

**工作量**：4-6小时

## 2. 因子有效性

### 2.1 [High] 回测与实时信号因子不一致

**问题描述**：
- 回测使用3个因子（`logit_prob`函数）：redeem_progress, premium_ratio, remaining_size
- 实时信号使用5个因子（`LogitScorer`类）：conversion_premium, trigger_progress, stock_momentum_5d, log_balance, is_triggered
- 两个系统的因子定义、权重、计算方式完全不同
- 回测结果无法反映实时信号的真实表现

**代码对比**：
```python
# backtest.py - 回测使用3因子
def logit_prob(redeem_progress, premium_ratio, remaining_size, weights):
    factors = [redeem_progress, premium_ratio, remaining_size]
    w = weights[:3] + [0.0, 0.0]  # 后两个因子权重为0

# signals.py - 实时使用5因子
z = (coeff["intercept"] + coeff["conversion_premium"] * x1 + ...)
```

**修复建议**：
1. 统一回测和实时信号的因子定义
2. 回测引擎应使用与实时信号完全相同的`LogitScorer`类
3. 将`logit_prob`函数废弃，统一使用`LogitScorer.score_single()`

**预期效果**：回测结果可反映真实策略表现

**工作量**：4-8小时

### 2.2 [High] 缺失2个关键因子 — stock_momentum 和 market_sentiment

**问题描述**：
- `stock_momentum`（正股动量）：回测中权重为0，signals.py中有实现但未在回测中验证
- `market_sentiment`（市场情绪）：两个模块都未实现
- 这两个因子在强赎博弈中非常重要：
  - 正股动量：正股上涨趋势会加速强赎触发
  - 市场情绪：转债市场整体热度影响溢价率

**代码位置**：`backtest.py:52-55`, `signals.py:271-275`

**修复建议**：
1. **stock_momentum**：使用正股5日/20日动量，从`get_stock_daily()`获取
2. **market_sentiment**：使用转债等权指数（如中证转债指数）的5日涨跌幅
3. 在回测中启用这两个因子，重新优化权重

**预期效果**：因子从3个增加到5个，模型解释力提升

**工作量**：6-8小时

### 2.3 [Medium] 因子计算方式不一致

**问题描述**：
- 回测中`premium_ratio`自行计算：`(cb_price - convert_value) / convert_value * 100`
- 实时信号中`conversion_premium`也自行计算：`(cb_price / conversion_value - 1) * 100`
- 两个公式数学等价，但数值精度可能不同
- 回测中`redeem_progress`使用`count/30`，实时信号使用`trigger_days_count/15`

**代码位置**：`backtest.py:87-92`, `signals.py:310-315`

**修复建议**：
1. 统一因子计算函数，放在`factors.py`中
2. 使用相同的分母（15天 vs 30天）— 实际强赎触发条件是15/30天

**预期效果**：消除因子计算差异

**工作量**：2-4小时

## 3. 回测框架

### 3.1 [Critical] 回测设计存在严重偏差

**问题描述**：
回测引擎的核心逻辑存在多个致命问题：

1. **使用当前数据回测历史**：已在上文1.1中描述
2. **价格匹配逻辑脆弱**：`_find_entry`方法在历史日线中找价格差异≤3%的日期，但：
   - 如果转债在回测日期前已经上涨/下跌，可能找不到匹配价格
   - 如果找到多个匹配日期，选择第一个，但可能不是正确的买入时机
3. **无交易成本**：未考虑佣金、印花税、滑点
4. **无流动性约束**：未检查成交量是否足够支撑仓位

**代码位置**：`backtest.py:148-240`

**修复建议**：
1. **重构回测引擎**：改为逐日遍历模式
2. **增加交易成本**：佣金0.03%，滑点0.1%
3. **增加流动性检查**：日成交量需大于仓位金额的10倍
4. **增加价格匹配的容错机制**：使用日期索引而非价格匹配

**预期效果**：回测结果更接近真实交易

**工作量**：24-32小时

### 3.2 [High] 无风险管理和仓位管理

**问题描述**：
- 固定仓位大小（20,000元/只），不考虑账户总资金
- 无最大回撤控制
- 无相关性控制（可能同时持有多个同行业转债）
- 无再平衡机制

**代码位置**：`backtest.py:110-115`

**修复建议**：
1. 实现凯利公式或固定比例仓位管理
2. 增加行业集中度限制（单行业≤30%）
3. 增加最大回撤止损（账户级-15%清仓）
4. 实现定期再平衡（每周/每月）

**预期效果**：降低回撤，提高夏普比率

**工作量**：8-12小时

### 3.3 [Medium] 绩效指标不完整

**问题描述**：
`calc_performance`函数只计算了基础指标：
- 缺少：夏普比率、最大回撤、卡玛比率、胜率-盈亏比、信息比率
- 缺少：不同市场环境下的分阶段表现
- 缺少：统计显著性检验

**代码位置**：`backtest.py:260-285`

**修复建议**：
1. 增加完整的绩效指标计算
2. 增加滚动回测（walk-forward analysis）
3. 增加蒙特卡洛模拟评估稳定性

**预期效果**：更全面的策略评估

**工作量**：4-6小时

## 4. 优化闭环

### 4.1 [High] 过拟合风险极高

**问题描述**：
优化器在**同一数据集**上搜索最优参数：
1. 使用当前快照数据回测
2. 搜索5个权重 + 3个阈值 = 8个参数
3. 迭代50次，每次评估不同参数组合
4. 选择"最优"参数
5. 无样本外验证

这相当于在训练集上反复调参，必然过拟合。

**代码位置**：`optimizer.py:180-250`

**修复建议**：
1. **时间序列交叉验证**：将数据分为训练期和验证期（如80%/20%）
2. **滚动窗口验证**：每6个月滚动一次，用前6个月训练，后1个月验证
3. **参数稳定性检查**：检查最优参数在滚动窗口中的稳定性
4. **减少搜索空间**：固定部分参数，只优化关键参数

**预期效果**：参数泛化能力提升，实盘表现更稳定

**工作量**：12-16小时

### 4.2 [Medium] 评分函数主观性强

**问题描述**：
`_calculate_score`函数使用主观权重组合指标：
```python
# balanced模式
return win_rate * 0.4 + avg_return * 0.3 - max_dd * 0.2 + trades * 0.1
```
- 权重缺乏理论依据
- 不同评分模式（balanced/aggressive/conservative）结果差异大
- 未考虑统计显著性

**代码位置**：`optimizer.py:260-270`

**修复建议**：
1. 使用夏普比率作为主要评分指标
2. 增加统计检验（如t检验）判断参数是否显著优于基线
3. 使用帕累托前沿分析，而非单一评分

**预期效果**：更客观的参数评估

**工作量**：4-6小时

### 4.3 [Low] 优化结果未进行敏感性分析

**问题描述**：
找到"最优"参数后，未分析：
- 参数微小变化对结果的影响
- 哪些参数对结果最敏感
- 参数之间的交互效应

**修复建议**：
1. 增加敏感性分析：对每个参数±10%变化，观察绩效变化
2. 生成参数重要性排序
3. 可视化参数-绩效关系

**预期效果**：了解参数鲁棒性

**工作量**：3-4小时

## 5. 信号生成

### 5.1 [High] 实时信号与回测信号不一致

**问题描述**：
如2.1所述，实时信号使用`LogitScorer`（5因子），回测使用`logit_prob`（3因子）。这意味着：
- 回测优化的参数无法直接用于实时信号
- 实时信号的"概率"与回测的"概率"含义不同
- 无法验证实时信号的阈值设置是否合理

**修复建议**：
1. 统一使用`LogitScorer`作为唯一的评分引擎
2. 回测引擎直接调用`LogitScorer.score_single()`
3. 优化器直接优化`LogitScorer`的系数

**预期效果**：回测-实时信号完全一致

**工作量**：4-6小时

### 5.2 [Medium] 正股动量计算有延迟

**问题描述**：
`_calc_stock_momentum_from_df`使用最近5个交易日的收盘价计算动量，但：
- 未考虑除权除息对价格的影响
- 未使用复权价格
- 动量计算窗口固定，未自适应

**代码位置**：`signals.py:230-240`

**修复建议**：
1. 使用前复权价格计算动量
2. 增加多窗口动量（5日、10日、20日）
3. 考虑使用动量强度而非简单涨跌幅

**预期效果**：更准确的正股趋势判断

**工作量**：2-3小时

### 5.3 [Low] 信号等级划分过于粗糙

**问题描述**：
信号等级使用固定阈值（85%/60%/30%/10%），但：
- 未考虑市场环境变化（牛熊市阈值应不同）
- 未考虑转债自身特性（不同规模/评级的转债阈值应不同）
- 未使用动态阈值

**代码位置**：`signals.py:280-285`

**修复建议**：
1. 实现自适应阈值（基于历史分位数）
2. 增加转债分类（大盘/小盘、高评级/低评级）的差异化阈值
3. 增加信号置信区间

**预期效果**：信号更精准，减少误报

**工作量**：4-6小时

## 6. 架构质量

### 6.1 [Medium] 错误处理不完整

**问题描述**：
- `data.py`中的函数在失败时返回空DataFrame，但调用方未充分处理
- `backtest.py`中`_get_cb_history`失败时返回None，但调用方未检查
- 网络请求无重试机制
- 部分异常被静默吞掉（`logger.debug`而非`logger.error`）

**代码示例**：
```python
# backtest.py:180
except Exception as e:
    logger.debug(f"获取 {cb_code} 日线失败: {e}")
return None  # 调用方可能未处理None
```

**修复建议**：
1. 实现统一的错误处理装饰器
2. 增加指数退避重试机制（retry with exponential backoff）
3. 关键路径增加断言和类型检查
4. 区分可恢复错误和致命错误

**预期效果**：系统稳定性提升

**工作量**：6-8小时

### 6.2 [Medium] 性能瓶颈

**问题描述**：
- `batch_compute`使用串行循环处理每只转债
- `preload_stock_cache`虽然使用线程池，但文件缓存未设置过期时间
- 回测引擎每次运行都重新获取数据，无缓存
- `_get_redeem_data`的5分钟缓存太短

**代码位置**：`signals.py:130-145`, `backtest.py:170-185`

**修复建议**：
1. 实现LRU缓存装饰器
2. 增加文件缓存过期时间（如24小时）
3. 回测引擎增加数据预加载和缓存
4. 使用异步IO替代线程池（如果数据源支持）

**预期效果**：运行速度提升2-5倍

**工作量**：4-6小时

### 6.3 [Low] 代码可维护性问题

**问题描述**：
- `backtest.py`中`signal_rank`函数过长（~80行），包含数据清洗、特征工程、评分逻辑
- 硬编码的列名映射分散在多个文件中
- 缺少类型注解（部分函数）
- 缺少单元测试

**修复建议**：
1. 拆分`signal_rank`为多个小函数
2. 统一列名映射到`config.py`或单独的`schema.py`
3. 增加完整的类型注解
4. 为核心逻辑编写单元测试

**预期效果**：代码可维护性提升

**工作量**：8-12小时

## 总结

该策略代码库整体架构良好，模块化清晰，但存在**致命的数据偏差问题**（前视偏差、幸存者偏差），导致当前回测结果**不可信**。建议优先修复回测框架的核心缺陷，然后统一信号生成逻辑，最后进行参数优化和系统完善。

**当前状态**：❌ 不可用于实盘决策
**修复后预期**：✅ 可辅助决策，但需持续监控和优化
